Can You Tell Which AI Wrote That Code? A New Method Tries to Fingerprint LLM-Generated Programs

Jiaxun Guo, Ziyuan Yang, Mengyu Sun, Hui Wang, Jingfeng Lu, Yi Zhang

Researchers propose a method to identify not just whether code was written by an AI, but which specific AI model produced it—by separating what the code does from how it does it.

  • What they did — Researchers from Sichuan University developed DCAN, a system that identifies which AI model (ChatGPT, Claude, DeepSeek, or Qwen) generated a piece of code by separating task semantics from model-specific fingerprints. They tested the approach on 91,804 code samples across 4 programming languages.
  • Key result — DCAN successfully distinguished between code generated by different AI models using representation disentanglement, extracting stylistic fingerprints like naming conventions, nesting patterns, and code verbosity.
  • Why it matters — This enables practitioners to trace AI-generated code back to its source model, which is crucial for licensing compliance, security audits, and code provenance in software supply chains.

The Problem

Most existing research on AI-generated code focuses on a binary question: did a human or a machine write this? But in practice, that's often not enough. When a vulnerability surfaces, a licensing conflict arises, or an incident investigation begins, the relevant question is more specific: which AI model produced this code? [§1] This is the problem of LLM Code Source Attribution—figuring out whether a snippet came from ChatGPT, Claude, DeepSeek, Qwen, or another model.

The challenge is that code generated by different LLMs for the same task tends to look similar on the surface. Ask four models to implement a binary search in Python, and they'll all produce syntactically valid code following the same algorithmic logic. The differences are subtle: variable naming conventions, comment styles, structural organization, whitespace preferences. The authors call these "generative fingerprints"—stylistic variations introduced by differences in training data, architectures, and decoding mechanisms [§1].

Prior passive attribution methods have relied on handcrafted stylometric features or language-specific indicators, limiting their scalability across programming languages [§2]. Active methods like watermarking require access to the generation pipeline, making them impractical for forensic scenarios where you're analyzing code after the fact [§2].

What They Did

DCAN frames attribution as a representation disentanglement task [§1]. The core idea: any code snippet contains two types of information tangled together. There's the functional semantics dictated by the programming task ("implement a sorting algorithm"). And there's the stylistic fingerprints unique to the model that generated it—how it names variables, how deeply it nests structures, how verbose it is [§3.1].

Conventional detection methods learn representations dominated by task-dependent semantics, so subtle model-specific fingerprints get overshadowed [§3.1]. DCAN explicitly pulls these apart: it decomposes a code snippet's latent representation into a task-logic component and a stylistic-signature component, then trains the task components from different models solving the same problem to be similar. Whatever's left over is the model-specific signal [§3.1, Figure 2].

To evaluate this, the team built what they describe as the first large-scale benchmark for this problem: 91,804 code samples generated by four LLMs across four programming languages (Python, Java, C, and Go), collected under two settings—with and without comments [§1].

The Results

DCAN achieves "reliable attribution performance across diverse settings" [Abstract], evaluated across all four languages and both coding settings. The source-specific signals it extracts map onto concrete stylistic dimensions: code verbosity, lexical density, naming convention bias (camelCase vs. snake_case tendencies), structural depth, and what the authors call "generative distinctiveness" [Figure 1]. These aren't hand-engineered features—they emerge from the disentanglement process.

The benchmark covers only four LLMs and four languages—a small slice of the real world. Production code, with its dependencies, boilerplate, and human edits layered on top of AI suggestions, would present a much harder target. And the approach assumes you know the candidate set of models in advance; it's a closed-set classification problem, not open-world detection [§3.1].

Why It Matters

Think of this as the beginning of a chain-of-custody infrastructure for AI-generated code. Today, organizations can ask "did AI write this?" but not "which AI?"—and the second question is the one that matters for license compliance, security triage, and policy enforcement [§1]. The disentanglement approach offers a principled framework: rather than pattern-matching on surface features, explicitly separate what code does from how it's written. The dataset and implementation are publicly available [§1, footnote], making it a foundation others can extend to more models and languages.

Memex: Teaching AI Agents to Take Notes Instead of Forgetting

Zhenting Wang, Huancheng Chen, Jiayun Wang, Wei Wei

When AI agents tackle tasks requiring dozens or hundreds of steps, they inevitably run out of working memory—and the standard fix of summarizing past work throws away the very details they'll need later.

  • What they did — Accenture researchers developed Memex, a system that lets AI agents compress their working context into indexed summaries stored externally, then retrieve exact evidence on demand via index references. They trained agents to manage their own memory using reinforcement learning (MemexRL).
  • Key result — Memex agents achieved improved task success rates on long-horizon tasks while using significantly smaller working context, preserving decision-critical details that lossy summarization would have discarded.
  • Why it matters — This provides a principled alternative to sliding windows and periodic summarization for practitioners building agents that need to handle complex, multi-step workflows without losing critical evidence.

The Problem

LLM-based agents are increasingly asked to do complex, multi-step work: searching scientific literature, exploring code configurations, orchestrating multi-API business processes [§1]. These tasks can span dozens to hundreds of tool calls, each producing observations and reasoning traces that pile up in the agent's context window.

Context windows are finite, but agent trajectories keep growing [§1]. Think of an agent that found a critical API error message at step 12 but doesn't effectively use it at step 87, even though it's technically still in the prompt. Long prompts "make distant evidence harder to use even when it is still present" [§1].

The standard workarounds—truncation or running summaries—are "fundamentally lossy because they compress or discard past evidence itself" [Abstract]. Once a tool output is summarized away, the exact error codes or configuration details are gone. Logging everything to external memory and retrieving by semantic similarity sounds better, but "when memory consists of a large pool of noisy, near-duplicate fragments, retrieval becomes ambiguous" [§1]. Similarity search can't tell the agent which results deserve stable references versus which branches are dead ends [§1].

What They Did

Memex introduces what the authors call Indexed Experience Memory. The core idea: compress context without discarding evidence [§1]. As an agent works, it periodically runs a CompressExperience action that replaces a long stretch of tool calls with a compact indexed summary—essentially a table of contents [Figure 1]. The full, unabridged content gets stored externally under those indices. When the agent needs past evidence, it calls ReadExperience(Index I) to pull the exact archived content back [Figure 1]. Like a researcher's notebook: you write down where you found something so you can retrieve the original later.

Critically, memory operations aren't hardcoded rules—they're "first-class actions in the same decision space as environment tools" [§1]. The agent decides what to summarize, how to index it, and when to retrieve it. To train these behaviors, the team built MemexRL, a reinforcement learning framework with reward shaping tailored to memory usage under a context budget [§1]. The hard problem is credit assignment: a well-designed index might only pay off dozens of steps later, while "a locally plausible but poorly structured summary can silently derail downstream reasoning" [§1].

The Results

On long-horizon tasks requiring agents to "interleave planning and tool calls over dozens to hundreds of steps while revisiting fine-grained evidence long after it first appears" [§1], Memex agents improved task success while using significantly smaller working context [Abstract]. The theoretical analysis suggests the approach can preserve decision quality with bounded retrieval as history grows—though the authors are careful to note this describes a best-case regime, not guaranteed behavior [§1].

The agent must now learn not just how to solve tasks but how to manage its own memory—a credit assignment problem the authors flag as "distinctive and difficult" [§1]. The empirical evaluation doesn't include detailed ablation numbers, and RL training costs remain an open question.

Why It Matters

If you've built agents that degrade on long tasks, the pattern here is worth studying: instead of choosing between "remember everything" and "summarize lossy," give the agent a notebook with page numbers. The comparison to how human experts manage complex investigations—through structured notes and stable references rather than raw recall—points toward a design principle for agentic systems that goes beyond this specific implementation.

The Paradox of Multimodal Training: Why Text-Only Data Makes Vision Models See Better

Ruilin Luo, Chufan Shi, Yizhen Zhang, Cheng Yang, Songtao Jiang, Tongkun Guan, Ruizhe Chen, Ruihang Chu, Peng Wang, Mingkun Yang, Yujiu Yang, Junyang Lin, Zhibo Yang

Training a vision-language model on text-only reasoning data makes it pay more attention to images than training it on multimodal data does — and a new metric explains why.

  • What they did — Researchers developed AVAR-Thinker, which uses Visual-Anchored Reflection data synthesis to force vision-language models to constantly refer back to visual details during reasoning, rather than relying on memorized text patterns.
  • Key result — Training on text-only reasoning data made VLMs attend more to images than multimodal training data did — a counterintuitive finding that improved performance on visual reasoning benchmarks.
  • Why it matters — This provides a simpler, more effective path to better visual reasoning in AI systems — text-only reasoning data is cheaper and easier to generate than multimodal training sets.

The Problem

Training AI models to reason about images—solving geometry problems from diagrams, interpreting charts—typically follows a two-stage recipe: a "cold-start" phase where the model learns reasoning patterns from examples, then reinforcement learning to refine those skills.

But practitioners have noticed something strange. When you initialize these models with text-only reasoning data—math proofs, logic puzzles, no images at all—they perform better on subsequent visual reasoning tasks than when you initialize them with multimodal data that includes images [§1]. Several groups had observed this [§1], but nobody had a quantitative explanation for why.

The stakes are practical. If multimodal cold-start data doesn't help, teams are wasting compute generating image-text reasoning examples. Without understanding the mechanism, there's no principled way to fix it.

What They Did

The researchers built a diagnostic metric called the Visual Attention Score (VAS), which measures how much a model's attention focuses on image tokens versus system prompt tokens [§3.1]. High VAS means the model is actively looking at the image; low VAS means it's mostly attending to boilerplate instructions.

They computed VAS across ten 7B-parameter vision-language models and found a striking correlation: r = 0.9616 between VAS and reasoning performance across four benchmarks [§3.2, Figure 1a]. Models with low visual attention consistently underperformed, regardless of their training data.

The key finding: models cold-started with multimodal data showed attention distributions nearly identical to the base model—they didn't learn to look at images any harder. But text-only cold-started models showed 15–20% higher attention to visual features [§3.3]. The authors call this Lazy Attention Localization: multimodal cold-start training fails to shift attention toward visual tokens, while text-only initialization induces stronger visual grounding [§3.3].

To test causality, they designed training-free interventions that directly amplified attention to visual tokens at inference time—producing consistent 1–2% accuracy gains without any retraining [§4]. They then built a full training framework called AVAR that embeds "visual anchors" (explicit references back to the image) throughout generated reasoning chains and uses attention-guided training objectives during cold-start [§5].

The Results

AVAR achieved an average gain of 7.0% across seven multimodal reasoning benchmarks [Abstract]. The largest improvements came on MathVision (+12.2%), testing multi-step geometric reasoning, and HallusionBench (+8.8%), measuring robustness against visual hallucinations [§1]. Each component contributed incrementally— removing any one reduced performance [§1, §6].

The VAS metric was computed on only 200 samples across ten models—a small sample for broad conclusions [§3.1, Figure 1a]. The benchmarks are heavily math-focused, and the precise mechanism behind Lazy Attention Localization remains partially unexplained [§1]. All experiments used 7B-parameter models [§3.1].

Why It Matters

This gives practitioners two concrete tools. First, a diagnostic: compute VAS for your model and check whether it's actually attending to visual inputs. If the score is low, your model may be ignoring images regardless of what multimodal data you trained on [§3.2]. Second, an immediate intervention: the training-free attention reweighting requires zero additional compute [§4].

The deeper question is about data curation budgets. If what matters during cold-start is the quality of reasoning patterns rather than modality coverage—and those patterns transfer across modalities—then teams generating expensive multimodal training data may be solving the wrong problem [§3.3]. Text-only reasoning data is cheaper, more abundant, and according to this work, more effective at teaching models to actually look at what you show them.
